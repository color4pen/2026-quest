# タスク実行ワークフロー

以下の手順でタスクを実施してください。
リファクタリング、機能追加、コンテンツ実装など全般に対応。

## Step 1: ブランチ作成

`feature/{タスク名}` ブランチを作成:

```bash
git checkout -b feature/{タスク名}
```

## Step 2: 設計ドキュメント作成

`docs/changes/YYYYMMDD-{タスク名}/design.md` を作成:

```markdown
# {タスクタイトル}

## 種別
- [ ] リファクタリング
- [ ] 機能追加
- [ ] コンテンツ実装
- [ ] バグ修正
- [ ] その他

## 概要
何を、なぜ変更/追加するか

## 現状分析
- 関連ファイル・既存実装の調査
- 変更が必要な箇所の特定

## 設計方針
- 新規ファイル/クラス/関数
- 既存コードへの変更
- データ構造/インターフェース

## 実装計画
- Step 1: ...
- Step 2: ...
- Step 3: ...

## 影響範囲
- 変更されるファイル一覧
- テストへの影響

## 承認
- [ ] 設計レビュー完了
- [ ] 実装開始
```

## Step 3: ユーザー承認（設計レビュー）

設計ドキュメントを提示し、承認を得てから実装開始。

- ユーザーからの指摘があれば設計を修正し、再度承認を得る
- **指摘があった場合のみ** `docs/reviews/design/YYYYMMDD-{タスク名}.md` にレビュー記録を作成
- レビュー記録には指摘内容と解決方法を記載（LGTMのみの場合は不要）

## Step 4: 実装

- TodoWriteでタスク管理
- 小さな単位で変更
- 各ステップ後に型チェック (`npx tsc --noEmit`)

## Step 5: テスト

- 必要に応じてテスト追加
- `npm test` で全テスト通過確認

## Step 6: コミット・PR

- 変更単位でコミット
- design.mdは参照用に残す
- PRを作成してレビュー依頼

## Step 7: PRマージ後

- **コードベースレビューが起点の場合**: 元のレビューファイル（例: `docs/reviews/codebase/20260305-ddd-review.md`）のステータスを `✅ 対応済み` に更新し、PR番号を記載
- 実装レビューで指摘があった場合のみ `docs/reviews/impl/` に記録

## 注意事項

- **レビュー記録に「対応中」などの途中状態を書き込まない** - 完了時にのみ記録を更新する
- 設計レビュー記録（`design/`）は指摘があった場合のみ作成
- 実装レビュー記録（`impl/`）は指摘があった場合のみ作成
