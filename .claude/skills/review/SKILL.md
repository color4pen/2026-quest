# コードレビュースキル

PRや設計ドキュメントのレビューを実施します。

## レビュー観点

レビュー時は `docs/reviews/REVIEW_GUIDELINES.md` の観点に従って確認してください。

### 設計レビュー

- **互換性**: 既存データとの後方互換性、マイグレーションパス、データ消失リスク
- **UX**: エラー時のユーザーへの伝え方、適切なエラーメッセージ
- **アーキテクチャ**: 責務の分離（SRP）、既存設計との一貫性、不要な抽象化（YAGNI）
- **型定義**: 配置先の適切さ、既存型との整合性
- **依存関係**: ライブラリの必要性、型定義の有無、ライセンス
- **実装計画**: Stepの粒度、作業漏れ

### コードレビュー（PRレビュー）

- **セキュリティ**: 入力値検証、タイミング攻撃、秘密情報のハードコード、OWASP Top 10
- **テスト**: 正常系・異常系、エッジケース、未使用変数、テストの意図
- **コード品質**: 命名、コメント、重複、エラーハンドリング
- **パフォーマンス**: 不要な処理、N+1問題、メモリリーク

## レビュー実施フロー

### 引数が設計ドキュメントパスの場合

設計レビューを実施する。

1. 設計ドキュメントを読み込み、レビュー観点に沿って確認
2. 指摘事項があれば整理して報告
3. 指摘がある場合のみ `docs/reviews/design/YYYYMMDD-{機能名}.md` に記録
4. **重要**: 設計レビュー後は実装に進まず、ユーザーの指示を待つ

### 引数がPR番号の場合

コードレビュー（PRレビュー）を実施する。

1. `gh pr diff {PR番号}` で差分を確認
2. レビュー観点に沿って確認
3. 指摘事項があれば整理して報告
4. 指摘がある場合のみ `docs/reviews/impl/YYYYMMDD-{機能名}.md` に記録

## 指摘の優先度

- **必須修正**: マージ前に修正が必要
- **推奨修正**: 修正が望ましいが、マージをブロックしない
- **軽微**: 気になる点だが対応不要

## レビュー履歴の記録

**指摘がある場合のみ** 履歴ファイルを作成する。LGTMの場合は作成不要。

### 設計レビュー (`docs/reviews/design/`)

```markdown
# 設計レビュー: {機能名}

- **日付**: YYYY-MM-DD
- **対象**: `docs/changes/YYYYMMDD-{機能名}/design.md`
- **レビュアー**: {名前}
- **結果**: 要修正 / 却下

## 指摘事項

### 1. {タイトル}（必須修正 / 推奨修正 / 軽微）
{内容}

## 次のアクション
{内容}
```

### 実装レビュー (`docs/reviews/impl/`)

```markdown
# 実装レビュー: {機能名}

- **日付**: YYYY-MM-DD
- **PR**: #{PR番号}
- **レビュアー**: {名前}
- **結果**: 要修正 / 却下

## 指摘事項

### 1. {タイトル}（必須修正 / 推奨修正 / 軽微）
{内容}

## 次のアクション
{内容}
```
