# コードレビュー観点

このドキュメントはコードレビュー時に確認すべき観点をまとめたものです。

## 設計レビュー

### 互換性
- [ ] 既存データとの後方互換性は考慮されているか
- [ ] マイグレーションパスは明確か
- [ ] バージョン移行時にデータが消失しないか

### UX
- [ ] エラー時のユーザーへの伝え方は定義されているか
- [ ] エラーメッセージは適切か（技術的すぎないか）
- [ ] ユーザーが混乱する挙動はないか

### アーキテクチャ
- [ ] 責務の分離は適切か（SRP）
- [ ] 既存の設計思想と一貫しているか
- [ ] 不要な抽象化をしていないか（YAGNI）
- [ ] クラスが必要か、純粋関数で十分か

### 型定義
- [ ] 新しい型の配置先は適切か
- [ ] 既存の型との整合性はあるか

### 依存関係
- [ ] 追加ライブラリは必要最小限か
- [ ] ライブラリに型定義は含まれているか（@types が必要か）
- [ ] ライセンスは問題ないか

### 実装計画
- [ ] Step の粒度は適切か
- [ ] 漏れている作業はないか

## コードレビュー

### セキュリティ
- [ ] 入力値の検証は適切か
- [ ] タイミング攻撃への耐性（機密データの比較時）
- [ ] 秘密情報のハードコードは許容範囲か
- [ ] XSS / SQLインジェクション等のOWASP Top 10

### テスト
- [ ] 正常系・異常系のテストがあるか
- [ ] エッジケースはカバーされているか
- [ ] テストコードに未使用変数はないか
- [ ] テストの意図は明確か

### コード品質
- [ ] 変数名・関数名は意図を表しているか
- [ ] コメントは必要十分か
- [ ] 重複コードはないか
- [ ] エラーハンドリングは適切か

### パフォーマンス
- [ ] 不要な処理はないか
- [ ] N+1 問題はないか
- [ ] メモリリークの可能性はないか

## レビュー結果の記録

レビューは設計と実装でディレクトリを分けて記録する。

### ディレクトリ構成

```
docs/reviews/
├── REVIEW_GUIDELINES.md
├── design/                    # 設計レビュー（個別機能）
│   └── YYYYMMDD-{機能名}.md
├── impl/                      # 実装レビュー（PRレビュー）
│   └── YYYYMMDD-{機能名}.md
└── codebase/                  # コードベース全体のレビュー
    └── YYYYMMDD-{テーマ}.md
```

### 設計レビューテンプレート (`design/`)

```markdown
# 設計レビュー: {機能名}

- **日付**: YYYY-MM-DD
- **対象**: `docs/changes/YYYYMMDD-{機能名}/design.md`
- **レビュアー**: {名前}
- **結果**: LGTM / 要修正 / 却下

## 指摘事項

### 1. {タイトル}（必須修正 / 推奨修正 / 軽微）
{内容}

## 良かった点
{内容}

## 次のアクション
{内容}
```

### 実装レビューテンプレート (`impl/`)

```markdown
# 実装レビュー: {機能名}

- **日付**: YYYY-MM-DD
- **PR**: #{PR番号}
- **レビュアー**: {名前}
- **結果**: LGTM / 要修正 / 却下

## 指摘事項

### 1. {タイトル}（必須修正 / 推奨修正 / 軽微}）
{内容}

## 良かった点
{内容}

## 最終判定
{内容}
```

### コードベースレビューテンプレート (`codebase/`)

```markdown
# コードベースレビュー: {テーマ}

- **日付**: YYYY-MM-DD
- **レビュアー**: {名前}
- **対象**: コードベース全体（{観点}）

## 指摘事項

### 1. {タイトル}（必須修正 / 推奨修正 / 軽微）{ステータス}
{内容}

**対応**: {対応内容とPR番号}
```

## コードベースレビューの運用ルール

コードベースレビュー（`codebase/`）は継続的に更新される文書です。

### ステータス管理

各指摘事項にステータスを付与する:
- `🔄 対応中` - 設計中または実装中
- `✅ 対応済み` - PR がマージされた
- `📝 今後の課題` - 認識しているが対応予定なし
- （ステータスなし）- 未着手

### 更新タイミング

1. **新しい指摘を追加した時**: ステータスなしで記録
2. **設計・実装を開始した時**: `🔄 対応中` に更新
3. **PR がマージされた時**: `✅ 対応済み` に更新し、PR番号を記載
4. **対応を見送る時**: `📝 今後の課題` に更新し、理由を記載

### 指摘の追加

コードベースレビューで新しい指摘が出た場合:
1. 該当するレビューファイル（例: `20260305-ddd-review.md`）に追記
2. または新しいテーマのファイルを作成
